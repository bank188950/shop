# การออกแบบส่วนผู้ดูแลระบบ

## สถานะ

หน้าผู้ดูแลมีภาพอ้างอิงสำหรับ Login และ Inventory แล้วเมื่อ 20 กรกฎาคม 2569 ใช้เป็นแนวทางสำหรับ `/admin/login`, `/admin`, `/admin/products` และ `/admin/products/new` โดยไม่ใช้ layout ของหน้าร้านลูกค้ากับหน้าปฏิบัติการ

- การชำระเงิน: `รอชำระเงิน` หรือ `จ่ายแล้ว`
- สถานะรายการสั่งซื้อ: `รอชำระเงิน` → `รอตรวจสอบ` → `เตรียมของ` → `พร้อมส่ง` → `ส่งแล้ว`; `ยกเลิก` เป็นกรณีพิเศษ
- ตารางรายการสั่งซื้อแสดงสถานะงานตามลำดับข้างต้นแยกจากคอลัมน์การชำระเงิน และไม่ใช้ `จ่ายแล้ว` เป็นสถานะรายการสั่งซื้อซ้ำกับคอลัมน์ชำระเงิน
- ในหน้ารายละเอียดรายการสั่งซื้อ ตัวเลือกสถานะสัมพันธ์กับการชำระเงิน: เมื่อ `รอชำระเงิน` เลือกได้เฉพาะ `รอชำระเงิน` และ `ยกเลิก`; เมื่อ `จ่ายแล้ว` เลือกขั้นตอนงาน `รอตรวจสอบ`, `เตรียมของ`, `พร้อมส่ง` หรือ `ส่งแล้ว` โดยไม่แสดง `ยกเลิก`
- สี badge ใช้สม่ำเสมอทั้งตาราง รายละเอียดรายการสั่งซื้อ และรายชื่อลูกค้าในรอบส่ง: `รอชำระเงิน` ใช้แดงพาสเทล (ข้อความ `#BD3B35`, พื้น `#FFE6E3`), `รอตรวจสอบ` ใช้เหลืองพาสเทล (ข้อความ `#9C5127`, พื้น `#FFF0D5`), `เตรียมของ` ใช้ม่วงพาสเทล, `พร้อมส่ง` ใช้ฟ้าพาสเทล, `ส่งแล้ว` และ `จ่ายแล้ว` ใช้เขียวพาสเทล และ `ยกเลิก` ใช้แดงพาสเทล

## แนวทาง

ส่วนผู้ดูแลควรดูสงบ กระชับ และเน้นงานปฏิบัติการ รองรับ workflow ด้านสต็อก คำสั่งซื้อ ช่วงเวลาจัดส่ง และการชำระเงิน ไม่ใช่พื้นที่สำหรับการเล่าเรื่องแบรนด์

## Token

ใช้ brand token ร่วมจาก `design-user.md`:

- `primary #2E7D32` สำหรับ navigation, active state และสถานะสำเร็จ
- `secondary #E3F2FD` สำหรับพื้นผิวข้อมูลหรือรายการลำดับความสำคัญต่ำ
- `tertiary #FFF3E0` สำหรับพื้นผิวสถานะรอดำเนินการ/ต้องให้ความสนใจ
- `neutral #455A64` สำหรับข้อความประกอบและป้ายข้อมูล

ใช้พื้นผิวเนื้อหาสีขาวบนพื้นหลัง neutral ที่อ่อนมาก สงวนสีแดงไว้สำหรับสถานะทำลายข้อมูล/error และห้ามสื่อสถานะงานปฏิบัติการด้วยสีเพียงอย่างเดียว

## โครงสร้างที่วางแผนไว้

- Desktop: มี sidebar ด้านซ้ายแบบคงที่ โดยข้อความเมนูมีขนาด 16px ตามด้วยชื่อหน้า KPI card แบบกระชับ filter และ data table
- Tablet และ Mobile (`≤ 1024px`): ยุบ sidebar เป็นปุ่มเมนูทรงกลมใน top bar และเปิด navigation แบบ drawer จากซ้ายพร้อม backdrop; top bar แสดงรายการสั่งซื้อใหม่ สินค้าใกล้หมด และโปรไฟล์ผู้ดูแล โดยมี badge จำนวนเมื่อมีรายการต้องติดตาม ส่วน filter เรียงซ้อนตามพื้นที่ และ data table เลื่อนแนวนอนได้พร้อม row action ชัดเจน
- มุมมองข้อมูลต้องทำให้สถานะคำสั่งซื้อ สถานะการชำระเงิน จำนวนสต็อก และช่วงเวลาจัดส่ง อ่านได้ในทันที
- ใช้ตารางสำหรับข้อมูลปฏิบัติการ พร้อม footer pagination รูปแบบเดียวกันในหน้ารายการข้อมูลของ Admin; หลีกเลี่ยงกราฟตกแต่งที่ไม่ตอบคำถามด้านการทำงานโดยตรง
- Login: วางเนื้อหาไว้กึ่งกลางหน้า มีตราระบบ หัวข้อ และฟอร์มใน card สีขาว โดยปุ่มหลักใช้สีเขียวเข้ม
- Dashboard: แสดงหัวข้อหน้า KPI card และรายการงานที่ต้องดำเนินการในพื้นผิวสีขาว
- Inventory: แสดง KPI 4 รายการ ปุ่มเพิ่มสินค้า และตารางสินค้า พร้อม status chip ที่มีข้อความกำกับ
- Product insert: ใช้ form card เดี่ยว มีฟิลด์ชื่อสินค้า หมวดหมู่ ราคา จำนวนสินค้า จำนวนชิ้น จำนวนชิ้นต่อ (1 สินค้า) หน่วยสินค้า ค่าแจ้งเตือนสต็อกต่ำ (ค่าเริ่มต้น 5) รายละเอียด การอัปโหลดรูป และ checkbox เปิดใช้งาน
- Product categories: มีเมนูแยกถัดจากสินค้า สำหรับแสดง เพิ่ม แก้ไข และลบชื่อหมวดสินค้า; ฟอร์มมีชื่อหมวดสินค้าและ checkbox ระบุจำนวนชิ้นต่อ (1 สินค้า) โดยตารางแสดงชื่อหมวดสินค้า สถานะระบุจำนวนชิ้น และการจัดการ
- Product units: มีเมนูแยกถัดจากหมวดสินค้า สำหรับแสดง เพิ่ม แก้ไข และลบชื่อหน่วยสินค้า; ฟอร์มและตารางแสดงเฉพาะชื่อหน่วยสินค้าและการจัดการ และใช้เป็นตัวเลือกในฟอร์มสินค้า
- Orders: ใช้ filter วันจัดส่ง รอบส่ง สถานะ และคำค้น ตามด้วยตารางรายการสั่งซื้อที่อ่านยอด ชำระเงิน และสถานะงานได้ทันที; รองรับ checkbox เลือกรายการสั่งซื้อที่ชำระเงินแล้วทีละรายการหรือทั้งหมดเพื่อเปลี่ยนสถานะแบบกลุ่มผ่าน popup โดยรายการสั่งซื้อรอชำระเงินจะถูกปิดไม่ให้เปลี่ยนสถานะ; รายละเอียดรายการสั่งซื้อต้องแยกข้อมูลผู้รับ รายการสินค้า และ control อัปเดตสถานะอย่างชัดเจน
- Preparation: มีเมนู `เตรียมของ` อยู่ก่อนเมนู `รอบส่งวันนี้` แสดงเฉพาะออเดอร์ที่จ่ายแล้วและมีสถานะ `รอตรวจสอบ` ตามวันจัดส่ง รอบส่ง และจุดรับที่เลือก; แอดมินเลือกออเดอร์เพื่อสร้างรอบเตรียมของได้หลายรอบผ่านปุ่มสีน้ำตาล ระบบเปลี่ยนออเดอร์ที่เลือกเป็น `เตรียมของ` และสรุปยอดสินค้าเป็นจำนวนตามหน่วยสั่งซื้อพร้อมจำนวนชิ้นที่คำนวณจากข้อมูลสินค้า; แต่ละรอบดูรายการของลูกค้าแต่ละคนและนำออเดอร์ออกจากรอบได้ก่อนปิดรอบ; ปุ่มสีน้ำตาลสำหรับเปลี่ยนสถานะทั้งรอบใช้ได้เฉพาะ `พร้อมส่ง` และล็อกรอบที่พร้อมส่งแล้ว โดยไม่หักสต็อกซ้ำ
- Users: มีเมนูผู้ใช้งานอยู่ถัดจากเมนู `ภาพรวม` สำหรับจัดการข้อมูลลูกค้าที่สมัครจากหน้าร้าน โดยตารางไม่มีรูปภาพและแสดงชื่อลูกค้า เบอร์โทรศัพท์ LINE ID จุดรับสินค้า สถานะการใช้งาน และการจัดการ; ฟอร์มเพิ่มหรือแก้ไขใช้ข้อมูลเดียวกับการสมัครสมาชิกโดยไม่แสดงหรือแก้ไขรหัสผ่าน
- Dispatch today: แยกจากรายการสั่งซื้อ ใช้เป็นใบเตรียมของ/ส่งของ เลือกรอบเช้าหรือบ่ายผ่านการ์ดสีฟ้าและส้มที่แสดงเวลาจัดส่งโดยไม่แสดงเวลาปิดรับ แล้วรวมเฉพาะรายการสั่งซื้อที่ชำระเงินแล้วตามหน่วยจริงและจัดกลุ่มตามจุดรับสินค้า; หมายเหตุท้ายหน้าแสดงรอบที่เลือกในวงเล็บ; สถานะรายการสั่งซื้อแสดงข้างรายชื่อลูกค้าแต่ละคน เพราะอาจต่างกันในจุดรับเดียวกัน และปุ่ม `เลือกสถานะ` พร้อมไอคอนรายการเปิด popup เพื่อเปลี่ยนสถานะของทุกรายการสั่งซื้อในจุดรับนั้นพร้อมกัน โดยมีเฉพาะตัวเลือก `พร้อมส่ง` และ `ส่งแล้ว`
- Settings: เวลาเป็นการตั้งค่าภาพรวมร่วมกันทุกจุดรับสินค้า โดยรอบเช้าปิดรับ 08:00 และส่ง 09:00–10:00; รอบบ่ายปิดรับ 12:00 และส่ง 14:00–15:00; มี mockup สำหรับตั้งค่าข้อความเตือนและการเปิดหรือปิด popup รวมถึงข้อความโฆษณาได้สูงสุด 3 รายการและการแสดงโฆษณาบนหน้าร้านในอนาคต
- Banners: มีเมนูแยกก่อนตั้งค่า สำหรับเพิ่มได้หลายแบนเนอร์ แต่ละรายการกำหนดหัวข้อ อัปโหลดรูปภาพ และระบุการเปิดใช้งานได้ในหน้าฟอร์มแยกเหมือนสินค้า พร้อมตัวอย่างภาพ และตารางรายการแบบเดียวกับสินค้าที่แสดงสถานะเปิดหรือปิดเพื่อแก้ไขหรือลบแบบยืนยันก่อนใช้บนหน้าแรก โดยไม่แสดงชื่อไฟล์แก่ผู้ดูแล
- Pickup locations: แยกเป็นหมวดสำหรับแสดง เพิ่ม แก้ไข และลบชื่อสถานที่รับสินค้า โดยฟอร์มมีชื่อสถานที่และ checkbox เปิดใช้งาน; สถานที่ที่ปิดใช้งานจะยังเก็บไว้แต่ไม่แสดงเป็นตัวเลือกให้ลูกค้า และการลบต้องยืนยันก่อนทำรายการ
- Dashboard: เริ่มด้วยกราฟเส้นรายวันสำหรับเดือน ปี และจุดรับสินค้าที่เลือก และสลับยอดขายหรือจำนวนรายการสั่งซื้อได้; มี filter วันจัดส่ง รอบส่ง และจุดรับสินค้า (ค่าเริ่มต้นวันนี้) ซึ่งใช้กับ KPI และตารางด้านล่างทั้งหมด ตามด้วย KPI ที่กดไปทำงานต่อได้ ได้แก่ ยอดขายที่ชำระแล้ว จำนวนรายการสั่งซื้อ รอดำเนินการ และรอชำระเงิน, ยกเลิก (เลื่อนสู่ตารางรายการสั่งซื้อด้านล่าง); จากนั้นแสดงตารางสรุปรอบส่งแยกจุดรับและตารางรายการสั่งซื้อรอการชำระเงินและยกเลิกพร้อมสถานะ

## การโต้ตอบและการเข้าถึง

- Filter, ปุ่ม และ status chip ต้องมีข้อความกำกับ และข้อความในส่วนผู้ดูแลต้องมีขนาดอย่างน้อย 16px ยกเว้น badge
- ยืนยันก่อนทำรายการที่ทำลายข้อมูล และแสดงสถานะ loading, empty และ error
- รักษาพื้นที่กดอย่างน้อย 44px เมื่อทำได้ และต้องมี keyboard focus ที่มองเห็นชัด
- ใช้ตัวเลขแบบ tabular สำหรับราคา จำนวน และยอดรวมเมื่อฟอนต์ที่เลือกสนับสนุน

## ก่อนเริ่มพัฒนา

ภาพอ้างอิงสำหรับ Login และ Inventory ได้รับแล้ว ส่วน route ที่เหลือควรได้รับภาพอ้างอิงหรือขยายเอกสารนี้ด้วยคอลัมน์ตาราง คำศัพท์สถานะ บทบาทผู้ใช้ และสถานะ empty/loading/error ก่อนพัฒนา
