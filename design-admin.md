# การออกแบบส่วนผู้ดูแลระบบ

## สถานะ

หน้าผู้ดูแลมีภาพอ้างอิงสำหรับ Login และ Inventory แล้วเมื่อ 20 กรกฎาคม 2569 ใช้เป็นแนวทางสำหรับ `/admin/login`, `/admin`, `/admin/products` และ `/admin/products/new` โดยไม่ใช้ layout ของหน้าร้านลูกค้ากับหน้าปฏิบัติการ

- การชำระเงิน: `รอชำระเงิน` หรือ `จ่ายแล้ว`
- สถานะออเดอร์: `รอชำระเงิน` → `รอตรวจสอบ` → `เตรียมสินค้า` → `กำลังส่ง` → `ส่งแล้ว`; `ยกเลิก` เป็นกรณีพิเศษ
- ตารางออเดอร์แสดงสถานะงานตามลำดับข้างต้นแยกจากคอลัมน์การชำระเงิน และไม่ใช้ `จ่ายแล้ว` เป็นสถานะออเดอร์ซ้ำกับคอลัมน์ชำระเงิน

## แนวทาง

ส่วนผู้ดูแลควรดูสงบ กระชับ และเน้นงานปฏิบัติการ รองรับ workflow ด้านสต็อก คำสั่งซื้อ ช่วงเวลาจัดส่ง และการชำระเงิน ไม่ใช่พื้นที่สำหรับการเล่าเรื่องแบรนด์

## Token

ใช้ brand token ร่วมจาก `design-user.md`:

- `primary #2E7D32` สำหรับ navigation, active state และสถานะสำเร็จ
- `secondary #E3F2FD` สำหรับพื้นผิวข้อมูลหรือรายการลำดับความสำคัญต่ำ
- `tertiary #FFF3E0` สำหรับพื้นผิวสถานะรอดำเนินการ/ต้องให้ความสนใจ
- `neutral #455A64` สำหรับข้อความประกอบและป้ายข้อมูล

ใช้พื้นผิวเนื้อหาสีขาวบนพื้นหลัง neutral ที่อ่อนมาก สงวนสีแดงไว้สำหรับสถานะทำลายข้อมูล/error และห้ามสื่อสถานะงานปฏิบัติการด้วยสีเพียงอย่างเดียว

## โครงสร้างที่วางแผนไว้

- Desktop: มี sidebar ด้านซ้ายแบบคงที่ ตามด้วยชื่อหน้า KPI card แบบกระชับ filter และ data table
- Mobile: ใช้ navigation control แบบกะทัดรัด, filter เรียงซ้อน และ data table ที่เลื่อนแนวนอนได้พร้อม row action ชัดเจน
- มุมมองข้อมูลต้องทำให้สถานะคำสั่งซื้อ สถานะการชำระเงิน จำนวนสต็อก และช่วงเวลาจัดส่ง อ่านได้ในทันที
- ใช้ตารางสำหรับข้อมูลปฏิบัติการ หลีกเลี่ยงกราฟตกแต่งที่ไม่ตอบคำถามด้านการทำงานโดยตรง
- Login: วางเนื้อหาไว้กึ่งกลางหน้า มีตราระบบ หัวข้อ และฟอร์มใน card สีขาว โดยปุ่มหลักใช้สีเขียวเข้ม
- Dashboard: แสดงหัวข้อหน้า KPI card และรายการงานที่ต้องดำเนินการในพื้นผิวสีขาว
- Inventory: แสดง KPI 4 รายการ ปุ่มเพิ่มสินค้า และตารางสินค้า พร้อม status chip ที่มีข้อความกำกับ
- Product insert: ใช้ form card เดี่ยว มีฟิลด์ชื่อสินค้า หมวดหมู่ ราคา จำนวนชิ้นต่อไม้ จำนวนคงเหลือ รายละเอียด การอัปโหลดรูป และ checkbox เปิดใช้งาน
- Product categories: มีเมนูแยกถัดจากสินค้า สำหรับแสดง เพิ่ม แก้ไข และลบชื่อหมวดสินค้า; ฟอร์มมีชื่อหมวดสินค้าและ checkbox ระบุจำนวนชิ้นต่อไม้ โดยตารางแสดงเฉพาะชื่อหมวดสินค้าและการจัดการ
- Orders: ใช้ filter วันจัดส่ง รอบส่ง สถานะ และคำค้น ตามด้วยตารางออเดอร์รายรายการที่อ่านยอด ชำระเงิน และสถานะงานได้ทันที; รองรับ checkbox เลือกออเดอร์ที่ชำระเงินแล้วทีละรายการหรือทั้งหมดเพื่อเปลี่ยนสถานะแบบกลุ่มผ่าน popup โดยออเดอร์รอชำระเงินจะถูกปิดไม่ให้เปลี่ยนสถานะ; รายละเอียดออเดอร์ต้องแยกข้อมูลผู้รับ รายการสินค้า และ control อัปเดตสถานะอย่างชัดเจน
- Dispatch today: แยกจากรายการออเดอร์ ใช้เป็นใบเตรียมของ/ส่งของ เลือกรอบเช้าหรือบ่ายแล้วรวมจำนวนสินค้าตามหน่วยจริงและจัดกลุ่มตามจุดรับสินค้า พร้อมสถานะเตรียมของ ออกส่ง และส่งครบ
- Settings: เวลาเป็นการตั้งค่าภาพรวมร่วมกันทุกจุดรับสินค้า โดยรอบเช้าปิดรับ 08:00 และส่ง 09:00–10:00; รอบบ่ายปิดรับ 12:00 และส่ง 14:00–15:00; มี mockup สำหรับตั้งค่าข้อความเตือนและการเปิดหรือปิด popup รวมถึงคำโฆษณาและการแสดงโฆษณาบนหน้าร้านในอนาคต
- Banners: มีเมนูแยกก่อนตั้งค่า สำหรับเพิ่มได้หลายแบนเนอร์ แต่ละรายการกำหนดหัวข้อ อัปโหลดรูปภาพ และระบุการเปิดใช้งานได้ในหน้าฟอร์มแยกเหมือนสินค้า พร้อมตัวอย่างภาพ และตารางรายการแบบเดียวกับสินค้าที่แสดงสถานะเปิดหรือปิดเพื่อแก้ไขหรือลบแบบยืนยันก่อนใช้บนหน้าแรก โดยไม่แสดงชื่อไฟล์แก่ผู้ดูแล
- Pickup locations: แยกเป็นหมวดสำหรับแสดง เพิ่ม แก้ไข และลบชื่อสถานที่รับสินค้า โดยฟอร์มมีชื่อสถานที่และ checkbox เปิดใช้งาน; สถานที่ที่ปิดใช้งานจะยังเก็บไว้แต่ไม่แสดงเป็นตัวเลือกให้ลูกค้า และการลบต้องยืนยันก่อนทำรายการ
- Dashboard: แสดง KPI และข้อมูลยอดขายย้อนหลัง พร้อมสรุปยอดขายวันนี้แยกตามจุดรับสินค้าเพื่อเชื่อมไปยังหน้าสรุปรอบส่ง

## การโต้ตอบและการเข้าถึง

- Filter, ปุ่ม และ status chip ต้องมีข้อความกำกับ
- ยืนยันก่อนทำรายการที่ทำลายข้อมูล และแสดงสถานะ loading, empty และ error
- รักษาพื้นที่กดอย่างน้อย 44px เมื่อทำได้ และต้องมี keyboard focus ที่มองเห็นชัด
- ใช้ตัวเลขแบบ tabular สำหรับราคา จำนวน และยอดรวมเมื่อฟอนต์ที่เลือกสนับสนุน

## ก่อนเริ่มพัฒนา

ภาพอ้างอิงสำหรับ Login และ Inventory ได้รับแล้ว ส่วน route ที่เหลือควรได้รับภาพอ้างอิงหรือขยายเอกสารนี้ด้วยคอลัมน์ตาราง คำศัพท์สถานะ บทบาทผู้ใช้ และสถานะ empty/loading/error ก่อนพัฒนา
